## Static File Pipeline

Most web applications — even those that deal primarily with dynamic content — also need to serve static files. These are files that don't change based on user input or application logic. Common examples include images, CSS stylesheets, and JavaScript files. In an ASP.NET Core project, these files are stored in a special folder called `wwwroot` and are served to the browser as-is when requested.

### The wwwroot Folder

By default, ASP.NET Core will only serve files from the `wwwroot` folder. It deliberately does not serve files from any other folder in your project, and this is an intentional security decision. Exposing arbitrary folders from your application could allow users to access sensitive files such as configuration files, source code, or private data.

The `wwwroot` folder is typically deployed to production exactly as it exists during development — all files and subfolders included. So whatever you place in `wwwroot` during development will be publicly accessible in production, which is why it should only ever contain files that are safe to expose to the public.

### Using StaticFileMiddleware

To serve static files in ASP.NET Core, you add `StaticFileMiddleware` to your pipeline using the `UseStaticFiles()` extension method. No additional service registration is needed — just a single line in your middleware configuration:

```csharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.UseStaticFiles();

app.Run();
```

That single call is all it takes to enable static file serving for your entire application.

### How the Pipeline Handles a Request

When a request comes in, the ASP.NET Core web server hands it off to the middleware pipeline. `StaticFileMiddleware` is the first (and in this simple example, the only) component in the pipeline, and it immediately evaluates whether it can handle the request:

**If the requested file exists in wwwroot:**
The middleware locates the file, reads it from disk, and returns it directly as the HTTP response. The request never travels any further down the pipeline. This is an example of terminal middleware — it short-circuits the pipeline and handles everything itself.

For example, if you have an image saved at `wwwroot/moon.jpg` and a user requests `/moon.jpg`, the middleware finds the file and sends it back to the browser immediately.

**If the requested file does not exist:**
The middleware cannot handle the request, so it passes it along to the next component in the pipeline. But what if there is no next component? In that case, ASP.NET Core automatically adds a dummy terminal middleware at the very end of every pipeline. This built-in fallback middleware does one simple thing — it returns a `404 Not Found` response to the browser.

> **Tip:** If no middleware in the pipeline generates a response for a given request, ASP.NET Core will automatically return a `404` error response. You never have to handle this case manually.

### Why This Design is Clean

The beauty of this approach is its simplicity and clarity. Rather than having complex configuration rules that decide what gets served and what doesn't, you simply look at the pipeline and immediately understand what will happen for any given request. The `StaticFileMiddleware` either handles the request or steps aside — there is no ambiguity. And because the fallback 404 behavior is automatic, you don't need to write any code to handle missing files either.
