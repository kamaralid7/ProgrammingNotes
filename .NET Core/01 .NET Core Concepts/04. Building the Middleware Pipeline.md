## Building the Middleware Pipeline

You build up the middleware pipeline in ASP.NET Core by calling methods on `WebApplication` (which implements `IApplicationBuilder`). Each method call adds a new middleware component to the pipeline.

### UseMiddleware

The `UseMiddleware<T>()` method is the primary way to add a custom middleware class to the pipeline. You pass your middleware class as the generic type argument, and ASP.NET Core handles wiring it into the pipeline automatically.

```csharp
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.UseMiddleware<ExceptionHandlerMiddleware>();
app.UseMiddleware<RoutingMiddleware>();
app.UseMiddleware<AuthenticationMiddleware>();
app.UseMiddleware<EndpointMiddleware>();

app.Run();
```

ASP.NET Core also provides many convenient extension methods that wrap `UseMiddleware` internally, giving you a cleaner syntax:

```csharp
app.UseExceptionHandler();
app.UseRouting();
app.UseAuthentication();
app.UseAuthorization();
app.MapGet("/", () => "Hello World!");
```

### Order Matters

It's important to remember that the order in which you call methods on `IApplicationBuilder` defines the order in which the middleware will run in the pipeline.

> **Warning:** When adding middleware to the pipeline, always take care to consider the order in which it will run. A component can only access data created by middleware that comes **before** it in the pipeline.

For example, if you place `UseAuthentication` after your endpoint middleware, your endpoints will never have access to the authenticated user — because authentication hasn't run yet by the time the request reaches them.

A typical correct ordering looks like this:

```csharp
app.UseExceptionHandler();      // 1. Catch any unhandled exceptions
app.UseHttpsRedirection();      // 2. Redirect HTTP to HTTPS
app.UseStaticFiles();           // 3. Serve static files early (short-circuits if found)
app.UseRouting();               // 4. Match the request to a route
app.UseAuthentication();        // 5. Identify who the user is
app.UseAuthorization();         // 6. Check if the user has permission
app.MapControllers();           // 7. Execute the matched endpoint
```

Each piece of middleware in this chain relies on the one before it. Authorization, for instance, needs to know who the user is — so authentication must always run first.
