
In ASP.NET Core, middleware is a C# class that can handle an HTTP request or response. Middleware can:

- Handle an incoming HTTP request by generating an HTTP response
- Process an incoming HTTP request, modify it, and pass it on to another piece of middleware
- Process an outgoing HTTP response, modify it, and pass it on to another piece of middleware or to the ASP.NET Core web server

You can use middleware in a multitude of ways in your own applications. A piece of logging middleware, for example, might note when a request arrived and then pass it on to another piece of middleware. Meanwhile, a static-file middleware component might spot an incoming request for an image with a specific name, load the image from disk, and send it back to the user without passing it on.

The most important piece of middleware in most ASP.NET Core applications is the EndpointMiddleware class. This class normally generates all your HTML and JavaScript Object Notation (JSON) responses.


## The Middleware Pipeline

Each middleware component handles the request and passes it on to the next component in the pipeline. After a middleware component generates a response, it passes the response back through the pipeline. When it reaches the ASP.NET Core web server, the response is sent to the user's browser.

**Pipeline:** This arrangement — whereby a piece of middleware can call another piece of middleware, which in turn can call another, and so on — is referred to as a pipeline. Think of each piece of middleware as a section of pipe; when you connect all the sections, a request flows through one piece and into the next.

### Cross-Cutting Concerns

One of the most common use cases for middleware is handling cross-cutting concerns — aspects of your application that need to occur for every request, regardless of the specific path or resource requested:

- Logging each request
- Adding standard security headers to the response
- Associating a request with the relevant user
- Setting the language for the current request

In each of these cases, the middleware receives a request, modifies it, and passes it on to the next piece of middleware. Subsequent middleware can then use the details added by earlier middleware to handle the request appropriately.

### Authentication and Authorization Example

For example, the authentication middleware associates the request with a user. The authorization middleware then uses this detail to verify whether the user has permission to make that specific request.

- If the user **has permission**, the authorization middleware passes the request on to the endpoint middleware to generate a response.
- If the user **does not have permission**, the authorization middleware short-circuits the pipeline and generates a response directly — returning it to the previous middleware without the endpoint middleware ever seeing the request.

This is an example of the **chain-of-responsibility** design pattern.

**Terminal Middleware:** When a middleware component short-circuits the pipeline and returns a response, it is called terminal middleware.

### Bidirectional Flow

A key point is that the pipeline is bidirectional:

1. The request passes through the pipeline in one direction until a piece of middleware generates a response.
2. The response then passes back through the pipeline in reverse order, passing through each piece of middleware a second time.
3. Finally, the first piece of middleware passes the response back to the ASP.NET Core web server.
